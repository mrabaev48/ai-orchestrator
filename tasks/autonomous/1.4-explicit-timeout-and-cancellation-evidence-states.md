1. Feature summary
Сделать timeout/cancellation first-class состояниями evidence с безопасной post-timeout/post-cancel recovery логикой.

2. Relevant modules
- `packages/core/execution-state/**`
- `packages/execution/**` (timeouts, cancellation propagation)
- `packages/state/evidence/**`
- `packages/application/errors/**`

3. Existing behavior
Timeout/cancel исходы частично теряются или агрегируются в generic failure, что снижает diagnosability и безопасность повторов.

4. Proposed design
- Расширить статусы evidence: `timed_out`, `cancelled`, `cancellation_requested`, `compensation_pending`, `compensated`.
- Добавить structured errors:
  - `STEP_TIMEOUT` (with timeoutMs, boundary, elapsedMs)
  - `STEP_CANCELLED` (with requestedBy, requestedAt, propagationState)
- Фиксировать checkpoint metadata: in-flight operations, dedup key state, compensation requirements.
- Гарантировать, что retry после timeout проходит через dedup guard и policy revalidation.
- Обеспечить cancellation propagation через execution graph без нелегальных state transitions.

5. Files likely to change
- `packages/core/src/execution-state/run-step-status.ts`
- `packages/execution/src/timeouts/**`
- `packages/execution/src/cancellation/**`
- `packages/state/src/evidence/**`
- `packages/application/src/errors/**`

6. Risks
- Скрытые "zombie" операции после timeout.
- Broken transition graph при одновременном timeout+cancel.
- Неполная компенсация side effects в partial-failure случаях.

7. Test plan
- Unit: timeout фиксируется как `timed_out` с полным metadata.
- Unit: cancellation фиксируется как `cancelled`, включая propagation details.
- Integration: timeout -> retry path не создает duplicate side effects.
- Integration: cancel mid-transition не ломает state integrity.
- Regression: generic failure больше не поглощает timeout/cancel семантику.

8. Rollout / migration notes
- Additive state expansion без удаления legacy `failed`.
- Обновить downstream consumers/read-model под новые статусы.
- Подготовить dashboard/runbook обновления для операторов.

9. Recommendation
Выполнять только после 1.2 и 1.3, иначе нельзя доказуемо обеспечить безопасные retries и forensic reconstruction.
