1. Feature summary
Реализовать production-grade append-only `RunStepEvidence` с checksum verification для forensic-grade реконструкции выполнения.

2. Relevant modules
- `packages/core/evidence/**` (domain model и инварианты)
- `packages/state/**` (append-only store, checksum validation)
- `packages/execution/**` (emission evidence на transitions)
- `packages/application/**` (query/read-model)

3. Existing behavior
Evidence trail не гарантирует неизменяемость и криптографически проверяемую целостность записей на всех критичных шагах.

4. Proposed design
- Ввести `RunStepEvidence` contract:
  - `evidenceId`, `tenantId`, `projectId`, `runId`, `stepId`, `attempt`, `status`, `policyDecisionId`, `idempotencyKey`, `timestamps`, `payloadRef`, `checksum`, `prevChecksum`, `traceId`.
- Реализовать append-only API (`append`, `readByRun`, `readByStep`), без update/delete.
- Использовать детерминированный canonical serialization + SHA-256 checksum.
- Проверять checksum-chain при чтении и сигнализировать `EVIDENCE_INTEGRITY_VIOLATION`.
- Гарантировать запись evidence для start/success/failure/blocked transitions.

5. Files likely to change
- `packages/core/src/evidence/run-step-evidence.ts`
- `packages/state/src/evidence/**`
- `packages/execution/src/evidence/evidence-emitter.ts`
- `packages/application/src/evidence/**`

6. Risks
- Повышенная latency/стоимость записи из-за checksum-chain.
- Возможные проблемы совместимости при canonicalization изменений.
- Частичные записи при недотранзакционном storage.

7. Test plan
- Unit: append-only enforced (update/delete rejected).
- Unit: checksum-chain валидация проходит на корректных данных.
- Unit: integrity violation детектируется и репортится.
- Integration: full run reconstruction из evidence без двусмысленности.
- Regression: порядок событий стабилен при retry/replay.

8. Rollout / migration notes
- Флаг `runStepEvidenceV2` + dual-write период.
- Legacy evidence читать через compatibility adapter (`integrity=unverified`).
- План backfill только для metadata, без подделки исторических checksums.

9. Recommendation
Выполнять сразу после 1.1; это базовый носитель фактов для timeout/cancel/dedup доказуемости.
